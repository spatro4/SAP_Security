## 1. Beginner Concepts

Classic ECC security rests on four building blocks: the **User Master Record** (SU01, stored across `USR*` tables), **Roles** (PFCG), **Profiles** (generated by roles, or manually assigned technical profiles), and **Authorizations** (the actual value-filled instances of authorization objects). A user logs on, the kernel loads their authorization buffer from these assignments, and every subsequent `AUTHORITY-CHECK` in the code evaluates against that buffer.

**Why ECC security still matters in 2026**: even greenfield S/4HANA and BTP-heavy landscapes retain an ECC-style ABAP stack underneath for many core processes, and the authorization concept is identical. Architects are expected to reason fluently in both worlds.

## 2. Intermediate Concepts

The user master record (`USR02` for logon data, `USR04`/`UST04` for profile assignment, `AGR_USERS` for role assignment with validity dates) tracks not just *what* a user can do but *when* that access is valid - role assignments carry `from`/`to` dates, a frequently overlooked control point for time-boxed access (contractor accounts, temporary elevated access).

**User types** matter for both security and licensing: Dialog (interactive), System (RFC/background, no dialog logon allowed - critical control to prevent interactive logon with a service account), Communication (legacy RFC), Service (anonymous, shared - heavily restricted), and Reference (used only as an authorization template via `SU01`'s "Reference User" field, never logged into directly).

## 3. Advanced Concepts

**Reference users** deserve architect-level attention: they let you grant a "template" set of authorizations to many users (common in Fiori/portal scenarios where an anonymous or reference user carries baseline authorizations) without each user individually holding those roles - but this indirection is also an audit blind spot if not documented, since `SUIM` role-assignment reports on the acting user won't show the reference user's authorizations unless you specifically check the `SU01` reference user field.

**Parameter IDs** (`SU3`/`SPA/GPA`) auto-populate field defaults (like default plant or sales org) - not a security control, but frequently confused with one; never rely on parameter IDs to restrict access, only to improve UX.

## 4. Architect Level Concepts

At scale, ECC user administration must integrate with an **Identity Management** process - manual SU01 maintenance does not scale past a few hundred users. The architecture question is always: where does the system of record for identity live (HR system, Active Directory, a dedicated IdM/IGA tool, or SAP Cloud Identity Services), and how does it provision/de-provision into ECC (via ALE/IDoc `PPOME`-driven HR triggers, SPML/SCIM through IPS, or custom middleware)?

## 5. Internal Working

At logon, the kernel calls `SAPMSYST`-managed logon processing, validates the password (or SSO assertion) against `USR02`, checks user validity dates and lock status (`UFLAG` in `USR02`), then builds the authorization buffer from all valid role/profile assignments in `UST04`/`AGR_USERS`, respecting each assignment's validity window. This buffer is held in the user's session context (shared memory) for the duration of the session, which is exactly why role changes require re-login or explicit buffer reset (`SU56`) to take effect.

## 6. Real Production Examples

A global retail client discovered during a SOX audit that 40+ terminated employees' user IDs remained unlocked for up to 30 days post-termination because the HR-to-SAP deprovisioning IDoc process only ran weekly and silently failed for users with non-standard employee IDs. Remediation: a daily reconciliation job comparing HR active-employee extract against `USR02` active users, alerting security operations to any mismatch within 24 hours - now a standard control in that client's SOX narrative.

## 7. SAP Notes (Reference Only)

Consult current SAP Notes for user type restrictions on Service/Communication users, and for the specific support package levels affecting `SU01`/`SU10` mass user maintenance behavior - always verify against your release.

## 8. Best Practices

- Never allow interactive (Dialog) logon for System or Service user types.
- Enforce password policy via profile parameters (`login/min_password_lng`, `login/password_expiration_time`, etc.) rather than relying solely on organizational policy.
- Automate deprovisioning from an authoritative HR source; never rely purely on manual SU01 lock requests.
- Time-box every privileged or contractor role assignment using the validity dates on the role assignment tab.

## 9. Common Mistakes

- Treating parameter IDs as an access control (they are not).
- Leaving terminated users unlocked due to slow or failing deprovisioning integration.
- Assigning roles without validity end dates for genuinely temporary access.
- Forgetting reference users when auditing "who has access to X."

## 11. Hands-on Lab

Create a test user via `SU01` with a `to` date 1 day in the future, assign a role, verify access works today, then verify - after the date passes - that access is automatically revoked without any manual role removal.

## 12. Troubleshooting

| Symptom | Cause | Tool |
|---|---|---|
| User locked unexpectedly | Failed logon threshold reached | `SU01`, `RSUSR006` |
| Role assignment "not working" | Validity date not yet started / already expired | `PFCG` user assignment tab |
| RFC user can log on interactively | User type misconfigured as Dialog | `SU01` |

## 13. Audit Perspective

Auditors will request evidence of timely deprovisioning (SLA-based), a current inventory of Service/System users with justification, and confirmation that generic/shared credentials are not in use for identifiable individual accountability.

## 14. Performance Impact

Excessive parameter ID and memory-resident buffer size (very large numbers of authorizations per user) can slow logon processing at scale; keep role design lean and avoid assigning dozens of overlapping composite roles to a single user.

## 15. Security Risks

Shared/generic Service user credentials break individual accountability - if misused, you cannot attribute the action to a specific person, which is itself an audit finding independent of what the action was.

## 16. Architecture

User provisioning architecture should flow: **Authoritative Source (HR/IdM)** → **Identity Provisioning/Middleware** → **SAP User Master + Role Assignment**, with a reconciliation job as a safety net, not the primary control.

## 17. Decision Making

When deciding between a Reference User pattern and direct role assignment, choose Reference User only for high-volume anonymous/portal scenarios where individual accountability isn't required at the application layer (it's typically handled by an upstream authentication layer instead).

## 18. FAQs

**Q: Does locking a user immediately invalidate their active session?**
A: No - a user already logged on keeps their session until it naturally ends; locking prevents new logons, it does not forcibly terminate active sessions (use `SM04`/`AL08` to terminate sessions explicitly if immediate termination is required).
